# Code Review Fix：代码审查和修复

审查所有更改的文件以查找重用、质量和效率问题。修复发现的问题。

## 使用场景

当用户想要审查更改的重用、质量和效率问题时使用。在进行代码更改后、提交之前调用。

## 参数

- `focus`（可选）：审查的额外关注区域

## 阶段 1：识别更改

运行 `git diff`（如果有暂存的更改则使用 `git diff HEAD`）查看更改了什么。如果没有 git 更改，审查用户提到的或您在此对话中之前编辑的最近修改的文件。

## 阶段 2：并行启动三个审查代理

使用 Task 工具在单条消息中同时启动所有三个代理。将完整的 diff 传递给每个代理，以便它有完整的上下文。

### 代理 1：代码重用审查

对于每个更改：

1. **搜索可能替换新编写的代码的现有工具和辅助函数**。在代码库的其他地方寻找类似的模式——常见位置是工具目录、共享模块和更改文件相邻的文件。
2. **标记任何重复现有功能的新函数**。建议使用现有函数代替。
3. **标记可能使用现有工具的内联逻辑**——手写的字符串操作、手动路径处理、自定义环境检查、临时类型守卫，以及类似的模式是常见的候选。

### 代理 2：代码质量审查

审查相同更改中的 hacky 模式：

1. **冗余状态**：复制现有状态的状态、可以派生的缓存值、可以是直接调用的 observers/effects
2. **参数蔓延**：向函数添加新参数而不是泛化或重构现有参数
3. **轻微变化的复制粘贴**：应该用共享抽象统一的近重复代码块
4. **泄露的抽象**：暴露应该封装的内部细节，或破坏现有的抽象边界
5. **字符串类型代码**：在使用已经存在的常量、枚举（字符串联合）或品牌类型的地方使用原始字符串
6. **不必要的 JSX 嵌套**：添加没有任何布局值的 wrapper Boxes/elements——检查内部组件 props（flexShrink、alignItems 等）是否已经提供了需要的行为
7. **不必要的注释**：解释代码做什么的注释（良好的命名标识符已经做到）、叙述更改或引用任务/调用者——删除；只保留非显而易见的 WHY（隐藏约束、微妙的不变式、workarounds）

### 代理 3：效率审查

审查相同更改的效率：

1. **不必要的工作**：冗余计算、重复文件读取、重复网络/API 调用、N+1 模式
2. **错过的并发**：可以并行运行的独立操作却顺序运行
3. **热路径膨胀**：添加到启动或每请求/每渲染热路径的新阻塞工作
4. **反复的无操作更新**：在轮询循环、intervals 或无条件触发的事件处理器中的状态/store 更新——添加变化检测 guard，以便在什么都没改变时不会通知下游消费者。同样：如果 wrapper 函数接受 updater/reducer callback，验证它是否尊重相同的引用返回（或任何"无变化"信号）——否则调用者的 early-return no-ops 会被静默击败
5. **不必要的存在检查**：在操作前预检查文件/资源存在（TOCTOU 反模式）——直接操作并处理错误
6. **内存**：无界数据结构、缺失的清理、事件监听器泄漏
7. **过度广泛的操作**：只需要一部分时读取整个文件、过滤一个时加载所有项目

## 阶段 3：修复问题

等待所有三个代理完成。聚合他们的发现并直接修复每个问题。如果发现是误报或不值得处理，记录下来并继续——不要争论发现，只是跳过它。

完成后，简要总结修复了什么（或确认代码已经很干净）。
