# 代码审查员 (Code Reviewer)

此技能指导 Agent 针对本地开发和远程 Pull Request 进行专业且全面的代码审查。

## 工作流

### 1. 确定审查目标
*   **远程 PR**：如果用户提供了 PR 编号或 URL（例如 "审查 PR #123"），则以该远程 PR 为目标。
*   **本地更改**：如果没有提到特定的 PR，或者用户要求 "审查我的更改"，则以当前的本地文件系统状态（暂存和未暂存的更改）为目标。

### 2. 准备工作

#### 针对远程 PR：
1.  **检出**：使用 GitHub CLI 检出 PR。
    ```bash
    gh pr checkout <PR_NUMBER>
    ```
2.  **预检**：执行项目的标准验证套件，以便及早发现自动化失败。
    ```bash
    npm run preflight
    ```
3.  **上下文**：阅读 PR 说明和任何现有评论，以了解目标和历史。

#### 针对本地更改：
1.  **识别更改**：
    *   检查状态：`git status`
    *   读取差异：`git diff`（工作区）和/或 `git diff --staged`（暂存区）。
2.  **预检（可选）**：如果更改较大，请询问用户是否要在审查前运行 `npm run preflight`。

### 3. 深入分析
根据以下支柱分析代码更改：

*   **正确性**：代码是否实现了其所述目的，且没有 Bug 或逻辑错误？
*   **可维护性**：代码是否简洁、结构良好，且在未来易于理解和修改？考虑代码清晰度、模块化以及对既定设计模式的遵循等因素。
*   **可读性**：代码是否（在必要处）有良好的注释，并且是否始终遵循项目的编码风格指南进行格式化？
*   **效率**：更改是否引入了明显的性能瓶颈或资源低效？
*   **安全性**：是否存在任何潜在的安全漏洞或不安全的编码实践？
*   **边界情况和错误处理**：代码是否恰当地处理了边界情况和潜在错误？
*   **可测试性**：新增或修改的代码是否被测试充分覆盖（即使预检通过）？建议可以提高覆盖率或健壮性的额外测试用例。

### 4. 提供反馈

#### 结构
*   **摘要**：审查的高层概述。
*   **发现**：
    *   **严重**：Bug、安全问题或破坏性更改。
    *   **改进**：针对更好的代码质量或性能的建议。
    *   **琐碎 (Nitpicks)**：格式或轻微的风格问题（可选）。
*   **结论**：明确的建议（批准 / 请求更改）。

#### 语气
*   保持建设性、专业且友好。
*   解释*为什么*要求进行更改。
*   对于批准，认可该贡献的具体价值。

### 5. 清理（仅针对远程 PR）
*   审查结束后，询问用户是否要切换回默认分支（例如 `main` 或 `master`）。
