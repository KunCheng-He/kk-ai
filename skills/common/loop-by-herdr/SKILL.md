# loop-by-herdr：目标驱动的迭代循环

OpenCode 单次会话不支持以目标为条件的自动重试。通过 herdr 的多会话编排能力，可以实现"执行→审查→优化→重试"的循环模式。

## 核心循环结构

```
定义目标 → [执行 → 审查 → 通过? → 退出 | 不通过 → 优化 → 重试]
                                              ↑                    │
                                              └────────────────────┘
                                              (最多 N 轮)
```

## 前提

- 运行在 herdr 环境中（`HERDR_ENV=1`）
- 项目中配置了 `opencode` 命令

如需了解 herdr 会话管理细节（创建 tab、等待完成、读取输出等），加载 `herdr` skill。

## 循环流程

### 1. 确定目标

在开始前，明确三个要素：

- **任务**：每次循环执行什么？（如"调研产品 X 并生成报告"）
- **成功标准**：怎样的产出算通过？用可验证的 checklist 描述（如"README.md ≥ 150 行、assets/ 含 2 张截图、visual_blueprint.json 存在"）
- **最大轮数**：最多迭代几轮？默认 5

### 2. 执行

通过 herdr 启动一个**新**的 OpenCode 会话执行任务：

- 在独立 tab 中启动新会话（不能复用旧会话，旧会话已被上轮任务污染上下文）
- 发送任务描述，等待会话完成
- 具体操作用 herdr 命令完成，加载 `herdr` skill 获取操作细节

### 3. 审查

**不要只读 OpenCode 的摘要输出**，直接读取生成的文件，逐项对照成功标准：

- 文件存在性：`ls -laR <output_dir>/`
- 内容完整性：检查关键章节是否完整、附件是否生成
- 数据有效性：URL 不为空、来源标注清晰、数字无矛盾

用工具直接验证（bash、Read），不依赖会话摘要的自我判断。

### 4. 判断

- **全部通过** → 退出循环，汇报结果
- **有未通过项 且 未达最大轮数** → 进入第 5 步优化
- **已达最大轮数** → 退出循环，列出未通过项和已知根因

### 5. 优化

根据审查结果，定位根因并修改对应文件：

| 根因类型 | 改什么 | 例子 |
|----------|--------|------|
| 流程指令不清 | skill 的 SKILL.md | 某个步骤标记为"可选"但实际必需 |
| 工具链/权限不匹配 | agent 定义或任务分配 | explorer 无权访问 playwright-cli |
| 规则缺失或矛盾 | 项目 AGENTS.md | 数据源优先级未定义 |
| 任务描述歧义 | 下一次的 prompt | 未指定输出格式 |

优化要精准——**每次只改一个变量**，否则无法确定哪个改动产生了效果。

优化后回到第 2 步，启动新会话执行下一轮。

## 审查策略

成功标准必须**可自动化验证**。用脚本或简单命令检查，避免主观判断：

```bash
# 文件存在
test -f <dir>/README.md && echo "PASS" || echo "FAIL"

# 文件大小合理
test $(wc -c < <dir>/README.md) -gt 5000 && echo "PASS" || echo "FAIL"

# 关键章节存在
grep -q "## 竞品对比" <dir>/README.md && echo "PASS" || echo "FAIL"
```

每轮记录：通过项、失败项、失败根因、本次优化动作。

## 反模式

- **一轮改多个变量**：无法归因，白跑一轮
- **复用好旧会话**：上下文已被上轮任务污染，新任务行为不可预测
- **只看摘要不读文件**：OpenCode 的自我总结可能遗漏致命问题
- **审查标准主观**：写"报告质量好"不如写"README.md 含官方介绍/用户评价/竞品对比三个章节，每个章节 ≥ 10 行"
- **不加最大轮数**：无限循环浪费资源
