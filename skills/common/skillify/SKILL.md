# Skillify（技能化）

将当前会话的可重复过程捕获为可重用的技能。

## 使用场景

当您想要将可重复的过程捕获为可重用的技能时使用。在您想要捕获的过程结束时调用。示例：
- "使其成为一个技能"
- "捕获此工作流"
- "技能化"
- "创建技能"
- "保存这个流程"

## 参数

- `description`（可选）：您想要捕获的过程描述

## 运行模式

- **内联运行**：在当前对话中执行，适合需要中途用户引导的工作流
- **分叉运行**：作为独立子代理运行，适合自包含、无需中途输入的任务

## 您的任务

### 步骤 1：分析会话

在提问之前，分析会话以识别：
- 执行了什么可重复的过程
- 输入/参数是什么
- 不同的步骤（按顺序）
- 成功的产物/标准（例如不仅仅是"编写代码"，而是"一个 CI 完全通过的开放 PR"）
- 您在哪里纠正或引导了过程
- 需要哪些工具和权限
- 使用了哪些代理
- 目标和成功的产物是什么

### 步骤 2：采访用户

您将使用 question 工具来了解用户想要自动化什么。重要说明：
- 所有问题都使用 question 工具！永远不要通过纯文本提问。
- 每轮迭代直到用户满意为止。
- 用户总是有一个自由的"其他"选项来输入编辑或反馈——不要添加您自己的"需要调整"或"我提供编辑"选项。只需提供实质性选择。

**第 1 轮：高层确认**
- 根据您的分析建议技能名称和描述。让用户确认或重命名。
- 建议高层目标和技能的具体成功标准。

**第 2 轮：更多细节**
- 将您识别的高层步骤作为编号列表呈现。告诉用户您将在下一轮深入细节。
- 如果您认为技能需要参数，根据您观察到的内容提出参数。确保您理解有人需要提供什么。
- 如果不明确，请问此技能是应该在当前对话中内联运行还是分叉运行（作为具有自己上下文的子代理）。分叉更适合不需要中途用户输入的自包含任务；内联更适合用户想要中途引导的情况。
- 询问技能应该保存在哪里。根据上下文提供默认建议（特定于仓库的工作流 → 仓库，跨仓库个人工作流 → 用户）。选项：
  - **此仓库**（`.opencode/skills/<name>/SKILL.md`）— 用于此项目特定的工作流
  - **个人**（`~/.config/opencode/skills/<name>/SKILL.md`）— 跨所有仓库跟随您

**第 3 轮：分解每个步骤**
对于每个主要步骤，如果不是非常明显，请问：
- 此步骤为后续步骤产生什么？（数据、产物、ID）
- 什么证明此步骤成功了，我们可以继续？
- 是否应该让用户确认后再继续？（特别是对于不可逆的操作如合并、发送消息或破坏性操作）
- 是否有任何步骤可以独立并行运行？（例如同时发布到 Slack 和监控 CI）
- 技能应该如何执行？（例如始终使用 Task 代理进行代码审查，或为一系列并发步骤调用代理团队）
- 有什么硬约束或硬偏好？必须或不应该发生什么？

您可以在这里进行多轮提问，特别是如果有超过 3 个步骤或很多澄清问题。根据需要迭代。

重要提示：特别注意用户在会话中纠正您的地方，以帮助为您的设计提供信息。

**第 4 轮：最后的问题**
- 确认何时应该触发此技能，并建议/确认触发短语。（例如对于 cherry-pick 工作流，您可以说：当用户想要将 PR cherry-pick 到发布分支时使用。示例："cherry-pick 到发布"、"CP 此 PR"、"hotfix"。）
- 您还可以询问任何其他需要警惕的事情，如果仍然不清楚的话。

一旦有足够的信息就停止采访。重要提示：对于简单的过程不要问太多！

### 步骤 3：编写 SKILL.md

在第 2 轮用户选择的位置创建技能目录和文件。

使用此格式：

```markdown
---
name: {{技能名称}}
description: {{单行描述}}
---

# {{技能标题}}
技能描述

## 使用场景
{{何时应该触发此技能的详细描述，包括触发短语和示例用户消息}}

## 参数
- `$参数名`: 此输入的描述

## 目标
明确陈述此工作流的目标。如果您有明确定义的产物或完成标准，最好。

## 步骤

### 1. 步骤名称
在此步骤中要做什么。具体且可操作。适当时包括命令。

**成功标准**：始终包含此项！这表明步骤完成，我们可以继续。可以是一个列表。

...
```

**每步注释**：
- **成功标准**是每个步骤必需的。这帮助模型理解用户对其工作流的期望，以及它应该何时有信心继续。
- **执行**：`Direct`（默认）、`Task agent`（简单的子代理），或 `[human]`（用户执行）。如果不明确，则不需要指定。
- **产物**：此步骤为后续步骤产生的数据（例如 PR 号、commit SHA）。仅在后续步骤依赖时包含。
- **人工检查点**：何时暂停并让用户继续。包括不可逆操作（合并、发送消息）、错误判断（合并冲突）或输出审查。
- **规则**：工作流的硬规则。会话中的用户纠正尤其有用。

**步骤结构技巧**：
- 可以并发运行的步骤使用子编号：3a、3b
- 需要用户操作的步骤在标题中包含 `[human]`
- 保持简单技能简单 — 2 步技能不需要在每个步骤上注释

### 步骤 4：确认和保存

在写入文件之前，将完整的 SKILL.md 内容作为 markdown 代码块输出到您的回复中，以便用户可以查看。然后使用 question 工具询问确认，使用简单的问题如"这个 SKILL.md 看起来可以保存吗？"——保持问题简洁。

写入后，告诉用户：
- 技能保存的位置
- 如何调用：提及技能名称
- 他们可以直接编辑 SKILL.md 来优化它

## 示例

用户说："把刚才的代码审查流程做成一个技能"

**第 1 轮**：
- 建议名称：`code-review`
- 建议描述：代码审查工作流，自动检查代码质量、运行测试、生成审查报告
- 目标：确保代码符合项目规范且所有测试通过

**第 2 轮**：
- 步骤：1) 运行 lint → 2) 运行测试 → 3) 生成审查报告
- 参数：`target`（要审查的文件或目录）
- 运行模式：分叉运行（自包含任务）
- 保存位置：此仓库

**第 3 轮**：
- 步骤 1 产物：lint 结果
- 步骤 2 成功标准：所有测试通过
- 步骤 3 产物：审查报告文件路径

**第 4 轮**：
- 触发短语："审查代码"、"code review"、"检查代码质量"

## 常见问题

- **用户中途取消**：保存已确认的部分，告知用户可稍后继续
- **参数提取失败**：使用 question 工具询问用户具体值
- **步骤不明确**：请用户提供更多细节或示例
