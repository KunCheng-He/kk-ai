# 工作流设计技能

帮助用户判断：给定一个开发场景，应该用什么方案（内置 agent / 自定义 subagent / skill / 组合）来搭建工作流。

## 三层架构

这是 AI 编码工具工作流的通用模型：

```
工作流 = 主 Agent（通用） + Subagent（专用） + Skill（可复用片段）
```

### 主 Agent（必须通用）
- 职责：理解意图 → 分解任务 → 委派执行 → 汇总结果
- **不要自定义成"专家"**。主 agent 需要什么都听得懂，因为要协调不同专业的 subagent
- 用工具自带的主 agent 即可，零成本

### Subagent（专用临时工）
- 职责：在主 agent 对话中 fork 出独立上下文窗口，执行具体任务后返回结论
- 核心价值：隔离噪声、限制权限、匹配成本（小模型探索 / 大模型推理）
- 内置 subagent 覆盖 80% 场景，只在特定领域需要自定义
- 典型内置：explorer（只读探索）、worker（执行实现）、general-purpose（复杂多步骤）

### Skill（可复用知识包）
- 职责：给 agent 的 on-demand 说明书，agent 启动时不加载，匹配到任务才加载
- 核心价值：流程固化，不需要每次都重新教 agent 怎么做

## Subagent 与 Skill 的区别

| 维度 | Subagent | Skill |
|------|----------|-------|
| 独立上下文窗口 | 是 | 否，注入到当前 agent |
| 并行执行 | 是 | 否，顺序执行 |
| 适合 | 需要隔离噪声的任务 | 需要遵循固定流程/规范的任务 |
| 例子 | "帮我搜索所有数据库调用" | "发 PR 的标准流程" |

## 五项设计原则

### 原则一：信息不回流
Subagent 产生的中间信息（grep 结果、测试日志、lint 输出、构建错误）**不能回流到主对话**。只回流传结论和行动摘要。这是 subagent 对抗 context rot 的核心机制。

### 原则二：读多写少，先读后写
探索阶段用只读 subagent 并行收集信息，确定路径后**只让一个 agent 写代码**。多个 agent 同时写同一个文件 = 冲突地狱。

### 原则三：权限逐步收紧
```
主 agent: 可读可写，需要审批
探索 subagent: 只读，自动通过
执行 subagent: 可写，但只限于工作区
审查 subagent: 只读，自动通过
```

### 原则四：模型与任务匹配
不要把旗舰模型用在 grep 搜索上。探索用小模型，复杂推理用大模型。

### 原则五：Skill 用于流程固化，Subagent 用于任务隔离
每次判断场景时问自己：这件事需要"自己的上下文窗口"还是"一套清晰指令"？
- 需要隔离 → subagent
- 需要复用指令 → skill
- 两者都要 → subagent + 预加载 skill

## 分析框架

当用户描述一个工作流场景时，按以下步骤分析：

### 第一步：拆解任务
把用户的场景拆成独立步骤，标注每一步的性质（探索 / 实现 / 审查 / 流程 / ...）。

### 第二步：对每一步做判断
对每个步骤，依次回答：

1. **会产生大量中间噪声吗？**
   - 是 → 考虑 subagent 隔离
   - 否 → 可以在主 agent 中完成

2. **需要限制权限吗？**
   - 是 → subagent，并在建议中明确写"只读"或"工作区写"
   - 否 → 主 agent 或 skill 都可以

3. **这个流程需要多次复用吗？**
   - 是 → 考虑 skill 固化
   - 否 → 内联指令即可

4. **有现成的内置 agent 能做吗？**
   - explorer 能做 → 直接用，不自定义
   - worker 能做 → 直接用，不自定义
   - 都不行 → 考虑自定义 subagent

5. **多个步骤能并行吗？**
   - 能 → 拆成多个 subagent 并行执行
   - 不能 → 串行即可

### 第三步：输出建议
给出推荐方案，格式如下：

```
场景: [一句话描述]

推荐方案:
- 主 agent: [自带通用 / 需自定义]
- Subagent: [内置 explorer / 内置 worker / 自定义 xxx / 不需要]
- Skill: [需要 xxx skill / 不需要]

理由:
1. [基于原则 x 的解释]
2. [基于原则 y 的解释]

实施建议:
[具体的 agent 定义或 skill 指令框架]
```

## 反模式警告

以下情况应主动提醒用户：

- **过度设计 agent**：场景不需要 subagent 时不要建议加。内置 explorer + worker 已经很强
- **多个 agent 同时写文件**：应建议串行或通过共享任务列表协调（如 agent teams）
- **主 agent 被定制成专家**：限制主 agent 会降低协调能力
- **Skill 里塞了大量指令**：Skill 超过 500 行就太重，考虑拆成多个 skill 或升级为 subagent

## 从零搭建工作流的推荐路径

1. 先写 `AGENTS.md` / `CLAUDE.md`，定义项目级规则
2. 直接用内置 subagent，不加任何自定义，感受瓶颈
3. 发现"每次都说同样的话"→ 变成 skill
4. 发现"某个任务总需要特定工具/权限/模型组合"→ 定义自定义 subagent
5. 不要一开始就造一堆自定义 agent
