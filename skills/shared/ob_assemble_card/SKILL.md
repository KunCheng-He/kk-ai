# 装配卡片技能 (Assemble Card)

## 1. 核心定位

此技能负责知识的**最终装配**。它不是"把卡片拼成长文"，而是**把多张精炼后的输出卡片整合为一张结构完整、交叉引用准确、可以直接放进 Obsidian 知识库的知识卡片**。

**最重要的原则**：你要交付的是一张 Obsidian 原子化知识卡片，不是一篇博客文章。不需要 Hook 开头、不需要 Echo 结尾、不需要段落间过渡叙事。只需要：**完整的信息架构 + 正确的交叉引用 + 干净的格式**。

## 2. 前置条件（启动时检查）

启动本技能前，先做以下检查。任一条件不满足时，按"降级策略"处理，不要硬跑流程。

| 条件 | 检查方式 | 降级策略 |
|---|---|---|
| `../ob_refine_card/assets/author_voice.md` 可读取 | 读取该文件 | **必须**：若缺失，提示"人格资产是核心约束" |
| `../ob_refine_card/assets/style_guide.md` 可读取 | 读取该文件 | **必须**：若缺失，提示"风格手册是核心约束" |
| `assets/card_template.md` 可读取 | 读取该文件 | **必须**：若缺失，提示"卡片模板资产缺失，请检查 skill 安装完整性" |
| 项目目录存在 | `glob "06-AI共建/[项目名称]/"` | 若不存在 → 提示先创建项目（或确认项目名） |
| 项目内有 ≥ 1 张已精炼的输出卡片 | glob `输出卡片/*.md` | 若为 0 → 提示先调用 `ob_refine_card` 精炼至少 1 张输出卡片 |
| 风格审计清单来源明确 | 引用 `../ob_refine_card/assets/style_guide.md` 第十三节 | 不允许在 SKILL 中维护独立清单副本 |

**禁止**：
- 跳过三份核心约束（author_voice / style_guide / card_template）的加载
- 在没有已精炼输出卡片的情况下直接进入装配
- 添加 Hook 开头、Echo 结尾、段落过渡叙事——这是卡片不是文章

## 3. 执行流程

### 步骤一：加载约束与模板 (Load Constraints & Template)

1. **先加载 `../ob_refine_card/assets/author_voice.md`** — 人格注入层。保持 hkc 的声音。
2. **再加载 `../ob_refine_card/assets/style_guide.md`** — 格式规范层。
3. **加载 `assets/card_template.md`** — 卡片结构模板。确保输出格式严格符合模板骨架。
4. 将三份约束注入上下文，全程参考。

### 步骤二：全局整合 (Integrate)

1. 确定项目路径：`./06-AI共建/[项目名称]/`。
2. 读取所有已精炼的"输出卡片"。
3. **异常处理**：若项目目录或卡片不存在，提示用户确认路径。
4. **排序**：如果之前的骨架确定了结构顺序，严格按顺序排列；若无，则按照"定义 → 详细展开 → 对比/应用 → 参考"的逻辑流排序。
5. **去重**：多张卡片可能覆盖了重叠内容（如同一概念的定义在多处出现），合并重复部分，保留最完整版本。

### 步骤三：提炼一级结构 (Restructure for a Single Card)

多张输出卡片拼在一起时，会产生冗余的章节编号和过渡。这一步负责：

1. **统一章节层级**：将所有卡片的一级标题（`# N.`）合并为一个连续的章节序列。如果原本卡片 A 有 `# 1.` 和 `# 2.`，卡片 B 也有 `# 1.` 和 `# 2.`，整合后变成 `# 1.` 到 `# 4.`。
2. **检查编号连续性**：确保从 `# 1.` 开始，无跳号。
3. **合并重复小节**：如果两张卡片都讲了同一个子主题（如"四种循环类型"），合并它们的内容，保留最完整版本。
4. **统一 Frontmatter**：使用卡片模板的 frontmatter 格式，合并所有来源卡片的 tags（去重），填写 interlinks（见步骤四）。

### 步骤四：交叉引用验证 (Cross-links Audit)

这是知识卡片区别于文章的关键特征。卡片必须与其他卡片建立准确的链接。

1. **扫描已有卡片**：用 glob 扫描以下目录中的 `.md` 文件：
   - `01-万象火锅/`、`02-*/`、`03-*/` — 已归档的知识卡片
   - `06-AI共建/` 其他项目 — 可引用的输出卡片
2. **注入 interlinks**：在 frontmatter 的 `interlinks` 字段中，列出本卡片引用了哪些已有卡片（`[[卡片名]]` 格式）。
3. **验证反向引用**：正文中的 `[[链接]]` 是否指向了真实存在的文件？用 glob 逐一验证。
4. **补充缺失链接**：正文中提到的概念如果已有对应卡片但未链接，补充 `[[链接]]`。

### 步骤五：卡片审计 (Card Audit)

装配完成后，按以下清单逐项检查：

**Frontmatter 完整性**：
- [ ] `id` 格式正确（LOOP-xxx 或项目前缀-xxx）
- [ ] `title` 包含 ID + Emoji + 口语化标题
- [ ] `tags` 至少 2 个，与内容相关
- [ ] `nodeType` 正确（通常是"输出"或"知识"）
- [ ] `created` 和 `updated` 日期正确
- [ ] `source_cards` 列出所有参考的知识卡片
- [ ] `interlinks` 列出所有交叉引用

**结构完整性**：
- [ ] 章节编号从 `# 1.` 开始，连续无跳号
- [ ] 二级章节用 `## N.M` 格式
- [ ] 每 2-3 个信息单元有 Callout 框
- [ ] 对比关系用表格，表头有 emoji
- [ ] 没有 `---` 分割线代替过渡
- [ ] 没有"本文将..."、"第一幕/第二幕"等禁词

**交叉引用准确性**：
- [ ] 所有 `[[链接]]` 指向真实存在的文件
- [ ] `interlinks` 列表与实际正文中的引用一致
- [ ] 没有指向不存在文件的死链接

**人格审计**（参见 style_guide 第十三节）：
- [ ] 全文有"我"的判断（至少 3 处）
- [ ] 有中心比喻（仅概念讲解型卡片）
- [ ] 有态度词（至少 2 处）
- [ ] 信息边界标注清晰
- [ ] 没有编造的内容

**禁词扫描**（命中即改）：
- "本文将"、"综上所述"、"双刃剑"、"不仅...而且"、"随着...的发展"、"在...的背景下"、"第一幕/第二幕"、"希望本文"、"让我们拭目以待"

### 步骤六：写入最终卡片 (Write Final Card)

1. 组装后的完整卡片写入 `./06-AI共建/[项目名称]/输出卡片/LOOP-xxx 「Emoji 标题」.md`
2. **反馈**：提供文件路径，告知审计结果（通过了哪些检查、修正了什么）
3. **提示**：告知用户可以将此卡片移动到 `01-万象火锅/`（或其他目标目录），并分配最终的 KNOW-/TECH- ID

## 4. 交互示例

> User: 好了，这几张输出卡片都精炼完了，帮我装成最终卡片。
> Agent: 好的。我把 3 张输出卡片整合为一张。章节从 6 个合并为 4 个（去掉了重叠的定义部分）。交叉引用验证通过——所有 `[[链接]]` 都指向真实存在的文件，补充了 2 个遗漏的链接。Frontmatter 的 interlinks 字段已更新。人格审计通过。
> Agent: (生成 `LOOP-001 「🔄Loop Engineering」.md`) 最终卡片已写入 `输出卡片/`。你可以直接把它移到 `01-万象火锅/` 并分配 KNOW 编号。
