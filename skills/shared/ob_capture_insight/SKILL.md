# 捕捉洞察技能 (Capture Insight)

## 1. 核心定位

此技能用于知识生产链条的**最前端**：将模糊的想法、对话或阅读材料，转化为清晰、准确、结构化的**原子化知识卡片**。

## 2. 前置条件（启动时检查）

启动本技能前，先做以下检查。任一条件不满足时，按"降级策略"处理，不要硬跑流程。

| 条件 | 检查方式 | 降级策略 |
|---|---|---|
| 用户提供了探索话题或碎片信息 | 上下文是否有可处理的输入 | 若无 → 提示用户输入起点（问题、链接、一段文字均可） |
| 涉及事实性引用时 webfetch 可用 | 调用 webfetch 试一下。**注意区分**：事实性引用（年份、数据、论文、版本号）→ 必须验证；理论概念讨论（如"什么是闭包"）→ 不强制 webfetch | 若不可用 → 标注"无法联网验证"，事实性引用降级为"使用 AI 调研"，并在产出时明确信息边界 |
| 目标项目目录归属明确 | 询问或推断当前项目名 | 若不明确 → 先询问用户项目归属，提供 3 个选项：1) 创建新项目，2) 加入已有项目，3) 仅作闪念胶囊（写入 `00-闪念胶囊/`）。若用户选了新建项目且 Agent 已执行 Stage 0 创建了目录则直接使用 |
| `assets/atomic_card_template.md` 可读取 | 读取该文件 | 若不存在 → 提示"模板资产缺失，请检查 skill 安装完整性"，不要凭空生成 |

**禁止**：
- 跳过 webfetch 直接编造**事实性信息**（年份、数据、论文引用、版本号）
- 默认写入 06 之外的主知识库目录（01/02/03）—— 那是 01/02/03 自己的工作流范围

## 3. 执行流程 (The Loop)

### 阶段一：苏格拉底引导 (Clarification)
*   **触发条件**：用户的输入模糊、宽泛，或缺乏上下文（例如：“我想了解智能体”）。
*   **动作**：
    *   不要直接输出长篇大论。
    *   提出 1-2 个引导性问题，帮助用户聚焦（例如：“是想了解技术原理，还是应用场景？”）。
    *   **目标**：将问题收敛到一个具体的“知识点”。

### 阶段二：事实锚点验证 (Verification)
*   **触发条件**：对话中涉及具体的**年份**、**数据**、**人名**、**代码版本**或**论文引用**。
*   **动作**：
    *   👉 **必须调用 `webfetch` 工具**进行核实。
    *   严禁编造或模糊处理事实性信息。
    *   在最终输出中通过脚注 `[^1]` 形式标注来源。

### 阶段三：知识结晶 (Crystallization)
*   **触发条件**：
    1.  对话出现“语义饱和”（概念定义清晰、有背景、有结论）。
    2.  用户显式要求（“生成卡片”、“总结一下”）。
*   **退出规则**：若经过 3 轮苏格拉底追问后仍未收敛到可结晶的粒度，不再继续追问。提示用户：“看起来这个话题比较大，要不要先挑一个具体的方向开始？我这边可以帮你列几个切入角度。”
*   **动作**：
    1.  **读取模板**：加载 `assets/atomic_card_template.md`。
    2.  **提炼内容**：将上下文中的核心信息填入模板。
        *   `{{concept_name}}`：准确的概念名称。
        *   `{{one_liner}}`：**一句话核心观点**——这张卡片最本质的判断。读者只读这一行，就应该知道它在讲什么。
        *   `{{tags}}`：2-3 个核心分类标签。
        *   `{{content}}`：核心定义 + 关键特征 + 应用场景（保持 Markdown 格式）。
    3.  **写入文件**：
        *   目标路径：`./06-AI共建/[项目名称]/知识卡片/`。
        *   文件名：`[Concept Name].md`。
        *   **异常处理**：若目标目录不存在，先创建目录再写入。
        *   **项目归属强制**：若 `[项目名称]` 仍未确定，禁止默认写入 `Default_Inbox`——必须先按前置条件中的三选一流程（创建新项目 / 加入已有项目 / 闪念胶囊）与用户确认。
     4.  **反馈**：告知用户卡片已创建，并展示卡片摘要。

### 阶段四：更新已有卡片 (Update)

*   **触发条件**：
    1.  用户说 "修改之前那张" / "纠正" / "之前的理解有误" / "补充" / "更新卡片"。
    2.  精炼阶段审计发现上游卡片有问题，Agent 发起了回退重做。
*   **动作**：
    1.  **定位卡片**：根据用户描述或错误标记找到目标卡片文件。
    2.  **读取现状**：读取卡片当前完整内容，用一句话向用户复述当前卡片的核心观点，确认理解一致。
    3.  **收集修正**：与用户确认需要修正的内容——是概念定义错了？核心观点偏了？还是缺少背景信息？
    4.  **精度匹配**：
        *   局部修正（改一处定义）→ 直接 `edit` 对应段落
        *   重大修正（核心观点反转）→ 重写全文，保持原 `id`、`created` 日期，更新 `updated` 字段
    5.  **反馈**：告知用户卡片已更新，展示变更摘要。
*   **补写规则**：如果用户是对已有知识进行**拓展**（而非纠正），判断是应该修改原卡片还是新增卡片。原则：同一概念的不同层面 → 修改原卡片；新概念 → 新建卡片。

## 4. 交互示例
> User: 讲讲 RAG 技术。
> Agent (调用本 Skill): RAG (检索增强生成) 是个大话题。你是想了解它的基本原理，还是最近的 GraphRAG 进展？
> User: 基本原理。
> Agent: (解释原理...)
> User: 好的，这就很清楚了。
> Agent: (检测到语义饱和 -> 生成卡片 -> 写入 `RAG原理.md`) 已为你生成关于 RAG 原理的知识卡片。
