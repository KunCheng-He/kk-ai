# 最终成稿技能 (Compile Manuscript)

## 1. 核心定位

此技能负责知识的**交付与封装**。它不是简单的"复制粘贴"，而是要为文章穿上"礼服"，确保其具备良好的阅读体验和传播价值。

**最重要的原则**：你要交付的是一篇**hkc 写的个人笔记**，不是一篇技术博客。声音必须是他的声音。每一句话都要经得起"这真的是我写的吗"的拷问——没亲自用过的东西，不能写得像用过一样。

## 2. 前置条件（启动时检查）

启动本技能前，先做以下检查。任一条件不满足时，按"降级策略"处理，不要硬跑流程。

| 条件 | 检查方式 | 降级策略 |
|---|---|---|
| `../ob_polish_prose/assets/author_voice.md` 可读取 | 读取该文件 | **必须**：若缺失，提示"人格资产是核心约束，缺它写不出 hkc 的声音" |
| `../ob_polish_prose/assets/style_guide.md` 可读取 | 读取该文件 | **必须**：若缺失，提示"风格手册是核心约束，缺它无法审计" |
| `assets/manuscript_template.md` 可读取 | 读取该文件 | **必须**：若缺失，提示"成稿模板资产缺失，请检查 skill 安装完整性" |
| 项目目录存在 | `glob "06-AI共建/[项目名称]/"` | 若不存在 → 提示先创建项目（或确认项目名） |
| 项目内有 ≥ 1 张已润色的输出卡片 | glob `输出卡片/*.md` | 若为 0 → 提示先调用 `ob_polish_prose` 润色至少 1 张输出卡片 |
| 风格审计清单来源明确 | 引用 `../ob_polish_prose/assets/style_guide.md` 第十三节 | 不允许在 SKILL 中维护独立清单副本——已经统一到 style_guide.md |

**禁止**：
- 跳过三份核心约束（author_voice / style_guide / manuscript_template）的加载
- 在没有已润色输出卡片的情况下直接进入交付
- 在不通过风格审计时强行交付

## 3. 执行流程

### 步骤一：加载人格与模板 (Load Persona & Template)
*   **动作**：
    1.  **先加载 `../ob_polish_prose/assets/author_voice.md`** — 人格注入层。读完后，整篇文章的语言、判断必须符合这个声音。
    2.  **再加载 `../ob_polish_prose/assets/style_guide.md`** — 格式规范层。
    3.  **加载 `assets/manuscript_template.md`** — 成稿结构模板。确保输出格式严格符合模板骨架。
    4.  将三份约束注入上下文，全程参考。

### 步骤二：全局整合 (Integrate)
*   **动作**：
    1.  确定项目路径：`./06-AI共建/[项目名称]/`。
    2.  读取所有已润色的"输出卡片"。
    3.  **异常处理**：若项目目录或卡片不存在，提示用户确认路径。
    4.  **排序**：如果之前确定了结构顺序，严格按顺序排列；若无，则建议一个最佳逻辑流。

### 步骤三：首尾构建 (Head & Tail)
*   **原则**：开头一刀见血，结尾有重量感。参考 style_guide 第二节（开头方式）和第十一节（结尾方式）。
*   **撰写 Hook (开头)**：
    *   ❌ 禁止："本文将介绍..."、"让我们探讨..."、"旨在..."
    *   ✅ 比喻切入（仿 TECH-0023）：
        > "LLM 就像是一个被困在聊天窗口里的'缸中之脑'——它博古通今，却无法帮你运行一行代码。"
    *   ✅ 场景切入（仿 TECH-0024）：
        > "你让 AI'按照公司规范提交报销'，它给你的却是网上搜来的通用模板。"
    *   ✅ Callout 框反问（仿 TECH-0026）：
        > `[!QUESTION] 我们到底在干什么？` —— 先抛问题，再展开。
    *   **核心**：开头 3 句话内必须让读者知道"这篇文章要解决什么问题"。
*   **撰写 Echo (结尾)**：
    *   ❌ 禁止："希望本文对你有帮助"、"综上所述"、"让我们拭目以待"
    *   ✅ 向前看但保持怀疑（仿 TECH-0023）：
        > "MCP 不一定是终点，但它是我们通往 AGI 必经的那段路。就像 USB 可能会消失，但'连接'的需求永存。"
    *   ✅ 个人判断 + 行动呼吁（仿 TECH-0026）：
        > "我的建议是：从简单开始，先用 HTTP 直接请求，遇到 JS 渲染才升级。"
    *   ✅ 回扣开头的比喻，形成闭环（仿 TECH-0024）：
        > "它将 AI 的能力从'说'提升到了'做'，为构建真正可靠的工程化 AI 应用铺平了道路。"
    *   **核心**：结尾要给读者一个"带走的东西"——一个判断、一个行动建议、或一个值得记住的画面。

### 步骤四：拼接与润色 (Stitch & Polish)
*   **动作**：
    1.  把 Head + 正文卡片 + Echo 拼接为完整文章。
    2.  检查段落间的过渡是否自然（Transition）：严禁直接贴卡片，必须加过渡句。
    3.  统一标题层级（H1, H2, H3），检查编号连续性。
    4.  确保所有交叉引用和脚注完整。
    5.  确保每个大章节之间有逻辑递进（起点 → 深入 → 进阶 → 反思），不要用 `---` 分割线代替过渡文字。

### 步骤五：风格审计 (Style Audit)
*   **核心**：交付前最后一道关。加载并严格按 `../ob_polish_prose/assets/style_guide.md` 第十三节（13.1-13.5）执行。该节是风格审计清单的**唯一权威源**（本文件不再保留副本，避免失同步）。
*   **执行流程**：
    1.  重新阅读 `../ob_polish_prose/assets/style_guide.md` 第十三节
    2.  跑完 13.1-13.4 全部可执行检查
    3.  按 13.5 表格分类到"局部可修"和"需重写"
    4.  局部可修 → 改对应段落；需重写 → 整篇推倒，但保留"已沉淀的事实卡片"作为素材池

### 步骤六：交付 (Deliver)
*   **动作**：
    1.  生成最终 Markdown 文件。
    2.  **写入路径**：`./06-AI共建/[项目名称]/最终成稿-[项目名称].md`。
    3.  **反馈**：提供文件链接，并告知用户风格审计结果（通过了哪些检查、修正了什么、是否触发了整篇重写）。
    4.  询问是否需要归档旧卡片。

## 4. 交互示例

> Agent: 好的。我把三部分拼起来了。开头我仿照你 TECH-0023 的写法，用"缸中之脑"的比喻切入。结尾给了个人判断和行动建议。风格审计格式/语言/人格/诚实四层全部通过。
> Agent: (生成 `最终成稿-AI写作.md`) 最终稿已生成，请过目。
