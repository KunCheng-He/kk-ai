# 润色文字技能 (Polish Prose)

## 1. 核心定位
此技能负责知识的**肉体塑造**：将干瘪的"逻辑骨架"填充为血肉丰满、语言生动的"正文段落"。

**最重要的原则**：你不是在写技术博客，你是在替 hkc 写他的个人笔记。先成为他，再动笔。

## 2. 执行流程

### 步骤一：加载人格与约束 (Load Persona & Context)
*   **动作**：
    1.  **先加载 `assets/author_voice.md`** — 这是人格注入层。读完后，用 hkc 的视角思考。
    2.  **再加载 `assets/style_guide.md`** — 这是格式规范层。
    3.  读取指定的"输出卡片"或用户提供的文本。
    4.  **异常处理**：若文件不存在，提示用户确认路径或直接使用默认风格。

### 步骤二：逻辑扩写 (Expand)
*   **核心原则**：**忠实于骨架，自由予表达**。
    *   **忠实**：严禁随意更改骨架中的逻辑推导链条（A -> B -> C）。
    *   **自由**：在表达方式上，使用类比、隐喻、具体的行业黑话来增强可读性。
*   **风格注入**：扩写时同步应用 style_guide 约束：
    *   章节编号用 `# N.` 格式，不用"第一幕/第二幕"
    *   每 2-3 段插入 callout 框（NOTE / TIP / QUESTION / WARNING）
    *   需要对比的内容用表格呈现，表头加 emoji
    *   首次出现的关键术语加 `**粗体**`
    *   **强制使用第一人称"我"分享观点和判断** — 参考 author_voice.md 三板斧
*   **输入示例**：`【AI降低门槛 → 垃圾内容泛滥 → 筛选成本上升】`
*   **输出示例**：
    > "AI 的确把创作门槛降到了地板上，但代价是互联网瞬间被生成的废话淹没。于是，原本用于创作的精力，现在被迫转移到了'从垃圾堆里找金子'上——筛选机制变得前所未有的昂贵。"

### 步骤三：风格审计 (Style Audit)
*   **核心**：基于 `assets/style_guide.md` 第十二节的自查清单，逐条审计输出文本。
*   **必须通过的检查项**：

    **格式层**：
    1.  标题有 emoji 且口语化？→ 若无 emoji，补上
    2.  章节用 `# N.` 数字编号，无"第一幕/第二幕"
    3.  callout 框至少 3-5 个/千字
    4.  该对比的地方用了表格
    5.  代码块语言标签完整，代码可运行

    **语言层**：
    6.  开头是"本文将..."吗？→ 若是，重写
    7.  有"综上所述""双刃剑""不仅...而且""随着...的发展"吗？→ 全删
    8.  结尾是"希望本文对你有帮助"吗？→ 重写
    9.  读起来像论文还是像聊天？→ 翻译腔改口语，长句拆短句

    **人格层（新增，最重要）**：
    10. **全文有"我"吗？** → 至少 3 处以上个人判断或经验，不是泛泛的"我认为"
    11. **有中心比喻吗？** → 概念讲解型至少 1 个，实操/推导类可跳过
    12. **有血泪教训吗？** → 至少 1 处"试了 X → 翻车 → 用 Y 解决"的具体故事
    13. **有态度吗？** → 至少 2 处明确的好坏判断，不是"各有利弊"端水
    14. **有"说人话"落地吗？** → 每个抽象概念后都有口语化翻译

### 步骤四：更新交付 (Update)
*   **动作**：
    *   直接在原文件中更新内容，或者在对话框中输出供用户确认。

## 3. 交互示例
> User: 帮我润色这段关于 Agent 的描述。
> Agent (加载 Author Voice + Style Guide): 现在的版本不像你——没有"我"的判断。我试着重写了，加了两处你可能会有的实操感受，去掉了端水句式。你看看这个版本有没有你的味道？
