# 成稿风格手册 (Style Guide for AI Writing)

> **使用前**：先加载 `author_voice.md`。这本手册定义的是格式规范；声音卡定义的是人格。两者一起用，缺一不可。

---

## 一、标题

**格式**：`[ID] 「[Emoji] [口语化标题]」`

| ✅ hkc 的真实标题 | ❌ AI 容易写成的标题 |
|---|---|
| TECH-0023 「MCP：让 AI 长出手👋」 | "深度解析 MCP 协议的技术原理与应用" |
| TECH-0026 「让 AI 成为一个合格的网民」 | "AI Agent 浏览器数据获取方案全景分析" |
| KNOW-0025 「🐎Harness Engineering」 | "Harness Engineering：LLM 可靠性工程体系研究" |
| TECH-0008 「🕋R5C 折腾日记」 | "NanoPi R5C 软路由配置指南" |

**规则**：
- Emoji 放在中文标题前面，不要放最后
- 用比喻、口语、或者直接用核心概念的一个生动翻译
- 不要学术化、不要"从 A 到 B"、不要"全景/深度/全面"

## 二、开头方式

| ✅ hkc 的开头 | ❌ AI 常见的开头 |
|---|---|
| "我的三个问题：1. Harness Engineering 是什么东西？2. 解决了什么问题？3. 为什么在 2026 年那么火？" (KNOW-0025) | "本文将系统介绍 Harness Engineering 的概念、应用场景与发展历程" |
| "LLM 就像是一个被困在聊天窗口里的'缸中之脑'。它博古通今，知道 Python 的 100 种写法，却无法帮你运行一行代码。" (TECH-0023) | "大语言模型在文本生成方面表现出色，但在实际操作层面存在明显局限" |
| "一个智能体的价值，不在于它能调用多少工具，而在于它如何从错误中学习。" (KNOW-0024) | "随着人工智能技术的快速发展，智能体已成为学术界和工业界的研究热点" |

**三种合法开头**：
1. **个人问题列表**："我的 N 个问题..."
2. **比喻切入**：用生活场景/常见事物做类比，一句话扎进去
3. **态度开场**：直接下一个判断，不铺垫

**开头要短、要扎人。一刀见血。禁止任何形式的"背景铺垫"。**

## 三、人称与声音

**这是最核心的一条。读三遍。**

| ✅ hkc 的写法 | ❌ AI 的写法 |
|---|---|
| "我的看法：视觉操控不是'贵所以不用'，而是'贵但在某些场景是唯一选择'" (TECH-0026) | "从成本效益的角度分析，视觉操控方案更适合特定场景" |
| "我把推动 Harness Engineering 爆发的力量拆成四个" (KNOW-0025) | "Harness Engineering 的兴起可以归纳为四个驱动因素" |
| "我也不知道为啥，参考网络上的办法，在系统-启动项中禁用 firewall" (TECH-0008) | "建议通过禁用防火墙服务来解决该问题" |
| "这玩意朴实得令人发指" (TECH-0023) | "MCP 协议的设计非常简洁高效" |
| "不建议通过 USB 口外挂移动硬盘，网络上说数据容易出问题，我暂时还没遇到，不过也不敢存重要数据" (TECH-0008) | "USB 外挂存储存在数据稳定性风险，建议谨慎使用" |

**铁律**：
1. 文中的每一个判断，都必须挂在一个具体的人身上——这个人叫"我"
2. "我们"仅用于带读者一起想问题时："你有没有想过..."、"我们到底在干什么？"
3. "你"直接对读者说话，像朋友聊天
4. **有态度，别端水**——说一个东西好就说它好在哪，烂就说它烂在哪
5. 允许且鼓励接地气的口语："朴实得令人发指"、"杀鸡用牛刀"、"折腾"、"这玩意"

## 四、说人话落地

每引入一个抽象概念或技术术语，紧接着用一句话把它"翻译"成人话。

| 抽象表述 | 你的人话翻译 |
|---|---|
| "Harness engineering 是围绕 LLM 构建评估、控制与可靠性基础设施的工程学科" | "说人话就是：你不信模型会做对，所以你用一套流程来确认真做对了——做错了就拦，拦不住就用兜底策略兜住，兜不住就报警让人来接管。" |
| "MCP 的底层通信协议是 JSON-RPC 2.0" | "别被 Protocol 这个词吓倒。所谓的协议就是传 JSON。" |

## 五、比喻贯穿（优先尝试，不硬塞）

概念讲解型笔记优先用**中心比喻**贯穿，把抽象概念锚定到具体生活场景。数学推导、实操日记类可以跳过。

hkc 用过的经典比喻：
- **USB-C** 比喻 MCP 的即插即用 (TECH-0023)
- **打电话 vs 派人去现场抄** 比喻 HTTP vs 浏览器 (TECH-0026)
- **厨房电器 vs 食谱** 比喻 MCP vs Skill (TECH-0024)
- **汽车**（引擎/方向盘/变速箱/仪表盘+刹车）比喻 AI 系统的各部分 (KNOW-0025)
- **拨号上网** 比喻未来看今天的浏览器方案 (TECH-0026)

## 六、血泪教训模式

**这是 hkc 最具辨识度的风格。** 不是"可能存在 X 问题"，而是"我遇到了 X、我试了 Y、翻了车、最后 Z 解决的"。

| ❌ AI 的泛泛提醒 | ✅ hkc 的血泪教训 |
|---|---|
| "在使用无头模式时，可能会遇到数据获取异常的问题，建议根据实际情况调整配置" | "尝试获取某网站的用户评论数据时，使用无头模式，获取的评论数据是空的。原因至今未明。但是经过尝试，使用有头模式可以正常获取。" |
| "旁路由配置中可能出现设备无法上网的情况，可参考相关文档进行排查" | "WIFI 设备无法上网...我只看懂了一半...以下是我在 H3C NX30Pro 里根据上面那篇博客执行的结果...最后我的解决办法如下" |

**规则**：血泪教训 = 具体场景 + 翻车过程 + 解决方案 + （可选）至今未解之谜

## 七、章节结构

### 编号规范
- 一级章节：`# N. [Emoji] 章节名`（N 为数字）
- 二级章节：`## N.M [Emoji] 小节名`
- **严禁**：第一幕/第二幕/第三幕（剧本式编号是 AI 味重灾区）

### 章节过渡
- **严禁用 `---` 分割线代替过渡**。每个大章之间要有一句逻辑递进的话。

| ❌ | ✅ |
|---|---|
| 两个章节之间直接放 `---` | "了解了如何控制输出后，我们来看看 LLM 最令人惊讶的特性——涌现能力。" |
| 章节间没有过渡句 | "现在我们有概念也有案例了。回到那个问题：为什么偏偏在 2026 年初爆了？" |

## 八、Callout 框使用规则

hkc 的标志性风格之一。**至少 3-5 个/千字。**

| Callout 类型 | 使用场景 | hkc 真实例子 |
|:---|:---|:---|
| `[!NOTE]` | 核心概念定义、关键结论 | "Harness engineering 是围绕 LLM 构建评估、控制与可靠性基础设施的工程学科" |
| `[!TIP]` | 实用技巧、反直觉经验 | "最轻量的方式——不启动浏览器，直接'打电话'要数据" |
| `[!QUESTION]` | 抛出核心矛盾 | "为什么是 MCP 杀出重围？" |
| `[!WARNING]` | 踩坑提醒、容易混淆 | "系统评估（第 2 层）是在上线前测的，约束控制（第 3 层）是每次推理时实时执行的" |
| `[!IMPORTANT]` | 强烈建议/反对 | "不建议通过 USB 口外挂移动硬盘" |
| `[!quote]` | 参考资料、外部引用 | 樟宜机场 IT 副总裁的原话 |
| `[!FAIL]` | 失败教训、反面案例 | "AI 不懂你的世界" |

**每 2-3 个段落必须出现一个 callout，但不要为了凑数硬塞。callout 必须承载有价值的内容。**

## 九、表格使用

hkc 大量用表格做对比。方案对比、优劣对比、概念区分——凡是能"左右对比"的，都用表格而不是大段文字。

| ✅ hkc 的典型表格 |
|---|
| 方案对比："HTTP 直接请求 vs Playwright vs WebBridge vs视觉操控" 的五行七列大表 |
| 概念区分："Prompt Engineering vs Agent vs Workflow vs MCP vs SKILL" 的五行三列表 |
| 层级结构："基准评测/系统评估/约束控制/生产监控" 的四行三列表 |
| 优缺对比："预剪枝 vs 后剪枝" 的五行六列表 |

**规则**：表头放 emoji，格式对齐。三行以上的对比，用表格不用列表。

## 十、代码与配置

1. 代码块必须指定语言：```python, ```bash, ```json, ```yaml, ```mermaid
2. 代码必须可运行，不写伪代码或省略号代码
3. shell 命令加行内注释，解释每步在干什么
4. 配置文件给完整的最小可运行版本
5. 代码前后有文字说明：这段代码解决什么问题、放哪里

## 十一、结尾方式

| ❌ AI 常见结尾 | ✅ hkc 的结尾 |
|---|---|
| "希望本文对您有所帮助" | "我的建议是：从简单开始，先用 HTTP 直接请求，遇到 JS 渲染才升级。" |
| "综上所述，Harness Engineering 是..." | "那座桥能撑多久不知道，但在桥塌之前，它就是最实用的过河方式。" |
| "随着技术的不断发展，让我们拭目以待" | "MCP 不一定是终点，但它是我们通往 AGI 必经的那段路。" |

**三种合法结尾**：
1. **向前看但怀疑**：承认当前的方案不是终极答案，但给出为什么现在它仍然是最好的
2. **回扣开头比喻**：回到第一段的那个画面，让它发生新的变化，形成闭环
3. **个人判断/行动建议**："我的建议是..."，给读者一个可以带走的判断

**结尾必须有重量感。宁可戛然而止，不要软绵绵的客套。**

## 十二、自查清单（AI 写完后的自我审查）

### 格式层
- [ ] 标题有 emoji 且口语化？
- [ ] 章节用 `# N.` 数字编号，而非"第一幕/第二幕"？
- [ ] 每 2-3 段有 callout 框？
- [ ] 该对比的地方用了表格？
- [ ] 代码块语言标签完整，代码可运行？
- [ ] 没有 `---` 分割线代替文字过渡？

### 语言层
- [ ] 开头是"本文将..."吗？→ 如果是，用比喻/问题/态度重写
- [ ] 有"综上所述""双刃剑""不仅...而且""随着...的发展"吗？→ 全删
- [ ] 有翻译腔长句吗？→ 拆成短句
- [ ] 结尾是"希望本文对你有帮助"吗？→ 重写

### 人格层（新增）
- [ ] **全文有"我"吗？** 至少有 3 处以上个人判断或个人经验
- [ ] **有中心比喻吗？** 概念讲解型笔记应有 1 个贯穿全文的生活化比喻（实操日记/数学推导可跳过）
- [ ] **有血泪教训吗？** 至少 1 处"我试了 X → 翻车了 → 用 Y 解决"的具体故事
- [ ] **有态度吗？** 有没有至少 2 处明确的好坏判断？
- [ ] **读起来像人说话还是像论文？** → 像人在说话
