# 精炼卡片技能 (Refine Card)

## 1. 核心定位

此技能负责知识的**结构化精炼**：将干瘪的"逻辑骨架"转化为由**表格、Callout 框、编号小节、交叉引用**组成的扫读友好格式。

**最重要的原则**：目标不是"读起来舒服的长文"，而是**扫一眼能找到东西的知识卡片**。你不是在写博客，你是在为 Obsidian 知识库组装一张可以独立存在、可以被搜索、可以被其他卡片引用的原子化笔记。

## 2. 前置条件（启动时检查）

启动本技能前，先做以下检查。任一条件不满足时，按"降级策略"处理，不要硬跑流程。

| 条件 | 检查方式 | 降级策略 |
|---|---|---|
| `./assets/author_voice.md` 可读取 | 读取该文件 | **必须**：若缺失，提示"人格资产是核心约束，缺它写不出 hkc 的声音" |
| `./assets/style_guide.md` 可读取 | 读取该文件 | **必须**：若缺失，提示"风格手册是核心约束，缺它无法审计" |
| 用户已指定输入 | 上下文是否有知识骨架文件或"输出卡片"路径 | 若无 → 询问用户：是精炼一张已有的"输出卡片"，还是直接给一段骨架 |
| 骨架来源可追溯（如果是精炼输出卡片） | 检查输出卡片是否有 `source_cards` 字段 | 若无 → 仍可精炼，但提示"该输出卡片缺失素材溯源，精炼时无法核对事实边界" |
| 风格审计清单来源明确 | 引用 `./assets/style_guide.md` 第十三节 | 不允许在 SKILL 中维护独立清单副本——已经统一到 style_guide.md |

**禁止**：
- 跳过 author_voice/style_guide 加载
- 在骨架空白的输出卡片上凭空精炼（应先调用 `ob_architect_structure`）
- 把"我"加进 AI 不知道 hkc 是否真说过的内容（这是诚实层的失败）
- 把知识卡片写成叙事文章——不需要 Hook、不需要 Echo、不需要段落间过渡

## 3. 执行流程

### 步骤一：加载约束与输入 (Load Constraints & Input)

1. **先加载 `./assets/author_voice.md`** — 人格注入层。用 hkc 的视角做判断。
2. **再加载 `./assets/style_guide.md`** — 格式规范层。
3. 读取指定的知识骨架文件或输出卡片。
4. **异常处理**：若文件不存在，提示用户确认路径。

### 步骤二：结构化重组 (Structure)

**核心原则**：忠于骨架的逻辑，改变表达的形式。骨架中的"核心观点 → 支撑案例 → 理论依据 → 延伸推论"链条不变，但表达方式从"段落叙事"切换为"表格 + Callout + 列表"。

**重组规则**：

| 原始骨架单元 | 重组为 | 示例 |
|---|---|---|
| 核心观点 | `[!NOTE]` Callout 框 | "Loop Engineering 的核心定义：让 AI Agent 重复执行工作周期..." |
| 对比关系（优劣/方案/概念区分） | 表格（表头加 emoji） | 六原语对比表、四种循环对比表 |
| 风险/踩坑/容易混淆 | `[!WARNING]` Callout 框 | 每个原语后的风险提醒 |
| 实用技巧/反直觉经验 | `[!TIP]` Callout 框 | "Automation 调用 skill 而非裸 prompt" |
| 实践案例/具体场景 | 列表 + 简短描述 | Osmani 的每日 loop 模板 |
| 延伸推论/个人判断 | 正文段落（保留"我"的判断） | "我的看法：这四层不是替代关系..." |
| 参考资料 | `[!quote]` Callout 框或尾部引用列表 | 来源链接 |

**格式约束**（严格遵循 style_guide）：
- 章节编号用 `# N.` 格式，二级用 `## N.M`
- 每 2-3 个信息单元必须出现一个 Callout 框
- 三行以上的对比，用表格不用列表
- 表头放 emoji，格式对齐
- 关键术语首次出现加 `**粗体**`
- 用"我"下判断、用比喻拆核心概念（只拆一个，不要每个子概念都配一个）

**诚实约束**：
- 没实操过的功能禁止写得像用过一样
- 不确定的数据不准写确定数字
- 信息边界要标注（亲自用过 → "我试了..."、看文档 → "据文档..."、AI 调研 → "使用 AI 调研..."）

**输出格式**：一段骨架章节 → 一个或多个结构化小节（表格 + Callout + 精炼段落）。不要用 `---` 分割线代替过渡文字。不要照搬骨架中的"过渡钩子"作为正文过渡句。

### 步骤三：交叉引用注入 (Cross-links)

1. 扫描 `./06-AI共建/[项目名称]/知识卡片/` 目录，找到所有知识卡片。
2. 也扫描 `01-万象火锅/`、`02-*/`、`03-*/` 目录中已有的 KNOW-/TECH- 卡片。
3. 在正文中，凡是涉及已有卡片覆盖的概念，插入 `[[卡片名]]` 链接。
4. 确保引用的卡片确实存在（glob 验证）。

### 步骤四：卡片审计 (Card Audit)

精炼完成后，做以下检查：

**结构完整性**：
- [ ] Frontmatter 包含 id、title、tags、nodeType、created、source_cards
- [ ] 每个骨架模块对应至少一个有内容的章节
- [ ] 章节编号（# N. / ## N.M）连续且格式正确

**格式一致性**：
- [ ] Callout 框类型使用正确（NOTE/TIP/WARNING/QUESTION/IMPORTANT/quote）
- [ ] 表格表头有 emoji，三行以上内容用了表格
- [ ] 没有 `---` 分割线代替文字过渡
- [ ] 没有"本文将..."、"综上所述"、"双刃剑"等禁词

**人格审计**：
- [ ] 全文有"我"的判断（至少 3 处）
- [ ] 概念讲解型笔记有 1 个中心比喻
- [ ] 有明确的好坏判断（至少 2 处态度词）
- [ ] 没有编造的内容，信息边界标注清晰

**禁词扫描**（命中即改）：
- "本文将"、"综上所述"、"双刃剑"、"不仅...而且"、"随着...的发展"、"在...的背景下"、"第一幕/第二幕"、"希望本文"、"让我们拭目以待"

### 步骤五：写入输出卡片 (Write)

**动作**：
1. 精炼后的内容写入 `./06-AI共建/[项目名称]/输出卡片/LOOP-xxx 「Emoji 标题」.md`
2. ID 编号逻辑：检查同目录现有卡片，按序递增
3. **反馈**：告知用户卡片已生成，提供文件路径，并报告审计结果

## 4. 交互示例

> User: 骨架已经有了，帮我精炼成卡片格式。
> Agent (加载 author_voice + style_guide + 骨架): 读了骨架的 6 个模块。我把对比关系都转成了表格，核心定义放进 NOTE callout，每个原语的风险用 WARNING callout 包裹。交叉引用了 KNOW-0024、KNOW-0025 和已有的 TECH 卡片。人格审计通过——有 5 处"我"的判断，中心比喻是"自动驾驶"。你看看有没有想要调整的？
> Agent: (生成 `LOOP-001 「🔄Loop Engineering」.md`) 卡片已写入 `输出卡片/`。
